Reroot DP

Created2023-10-11|Updated2023-12-30

Last CF contest, I solved A~C really quickly, but got stuck for over an hour on a rerooting dp problem. In this blog, I want to learn how to do reroot dp!

When to reroot dp?

(Disclaimer: I will refer uuu as the parent node, and v,cv, cv,c as the child node)
Reroot DP occurs when the problem wants a answer that would require making each node as the root of the tree.
You should be able to calculate one of the answers in maybe O(n)O(n)O(n) time, and is able to transition subtree/outside subtree informations in less than the time to construct them individually with the help of some information gathered during the calculation of the first answer (subtree information, depth…etc).

Lets check out a basic problem to understand reroot dp more:

LeetCode 834. Sum of Distances in Tree

The problem is basically: For every node iii, return the sum of depth if iii is the root of the tree.
The first step for reroot dp problems is to first determine the answer of a root, lets try to find the answer for node 000.
This is quite trivial, we denote sum[u]sum[u]sum[u] as the depth sum of subtree uuu, we can maintain depth[v]=depth[u]+1depth[v] = depth[u] + 1depth[v]=depth[u]+1 with depth[0]=0depth[0] = 0depth[0]=0, and sum[u]=depth[u]+∑sum[v]sum[u] = depth[u] + \sum sum[v]sum[u]=depth[u]+∑sum[v] with a simple dfs like this:

But when thinking of how to build the answer for the first problem, you also want to start thinking about what information that you need in the rerooting process that can be maintained in the first dfs.
In this problem, we can start thinking of the transition process, then we can know what we want to track in first dfs.

The numbers labeled in red is the depth when 000 is the root, and green is when 222 is the root. Notice the nodes inside the yellow circle (nodes that aren’t in the subtree of 222) all increased 1 depth, while the nodes inside the purple circle (nodes that are in the subtree of 222) all decreased 1 depth.

Another example is from 222 to 333, where green is the depth when 222 is the root, and blue is 333. You can also see the same transition. we can thus determine the dp transition between nodes:

Now, lets think of the transition in our second dfs:

Denote dp2[i]dp_2[i]dp2​[i] as the number of ways to color the tree after removing iiis subtree. This may seem pretty sudden, but it makes sense if you fully understood reroot dp (which I didn’t so I struggled on this question for a long time)

In a reroot dp problem, you can treat the whole tree as two components: the subtree of a node iii, and other parts of the tree. Since when you make iii the root instead, all the parts that didn’t belong to the subtree will then be inside the subtree (with the original parent node as its child), so if we can maintain the value for this newly added part, we can calculate the answer similar to how we got the first root (as we would have the values for all iiis child).

dp2[v]dp_2[v]dp2​[v] here basically means the dp1dp_1dp1​ we had, but for node uuu when vvv is the root instead.

Now, let’s actually solve the problem I was stuck in contest.

CF 1882D. Tree XOR

Let’s first think about how to obtain the answer for the root.
From our root, we can greedily change every child vvv into the value our root has with (val[u]⊕val[v])⊕subtree[v](val[u] ⊕ val[v]) ⊕ subtree[v](val[u]⊕val[v])⊕subtree[v].
We also need to remember that the nodes after the child ccc also changed to (val[u]⊕val[v])⊕val[c](val[u] ⊕ val[v]) ⊕ val[c](val[u]⊕val[v])⊕val[c] when calculating.
Let’s denote the value that we want to apply to the subtree[u]subtree[u]subtree[u] as xor_val[u]xor\_val[u]xor_val[u], which xor_val[root]xor\_val[root]xor_val[root] is 0.
calulating xor_valxor\_valxor_val during dfs is quite easy: xor_val[v]=(xor_val[u]⊕val[v])⊕val[1]xor\_val[v] = (xor\_val[u] ⊕ val[v]) ⊕ val[1]xor_val[v]=(xor_val[u]⊕val[v])⊕val[1].
But there is acutally a pretty nice observation here:

The part circled in yellow is the original value after the operation on node 222, and the red part is the value after node 444.
The value of every xor_val[v]xor\_val[v]xor_val[v] is just val[u]⊕val[v]val[u] ⊕ val[v]val[u]⊕val[v]!
This tells us the value to apply does not change with the root, which makes our lives much easier.
So, the first part of dfs should look like this:

and ans[root]=∑i=1nxor_val[i]⋅subtree[i]ans[root] = \sum_{i = 1}^{n} xor\_val[i] \cdot subtree[i]ans[root]=∑i=1n​xor_val[i]⋅subtree[i].

Now, for the rerooting part:

(xor_val[i]⋅subtree[i])(xor\_val[i] \cdot subtree[i])(xor_val[i]⋅subtree[i]) does not change with the root. The only changing ones are the current root (uuu), and the next child (vvv).

uuu will become the child of vvv, so xor_val[u]=val[v]⊕val[u]=xor_val[v]xor\_val[u] = val[v] ⊕ val[u] = xor\_val[v]xor_val[u]=val[v]⊕val[u]=xor_val[v], and the subtree size is just n−subtree[v]n - subtree[v]n−subtree[v].
For vvv, (xor_val[v]⋅subtree[v])(xor\_val[v] \cdot subtree[v])(xor_val[v]⋅subtree[v]) is just 000 since it’s the new root.
Combine them together, we have the following transition:

ans[v]=ans[u]+(xor_val[v]⋅(n−subtree[v]))−(xor_val[v]⋅subtree[v])ans[v] = ans[u] + (xor\_val[v] \cdot (n - subtree[v])) - (xor\_val[v] \cdot subtree[v]) ans[v]=ans[u]+(xor_val[v]⋅(n−subtree[v]))−(xor_val[v]⋅subtree[v])

Which equals
